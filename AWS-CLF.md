### 扩展 Amazon EC2
#### 1.可扩展性
可扩展性是指系统通过增加资源来处理更大负载的能力。您可以通过向现有计算机添加更多功能来实现纵向扩展，也可以通过增加更多的计算机来实现横向扩展。可扩展性侧重于长期的容量规划，确保系统容量能够按需增长，容纳更多用户或工作负载。

- 【横向扩展】增加实例个数
#### 2.弹性
弹性是指根据实时需求自动纵向扩展或缩减资源的能力。系统随后可以快速调整资源，在高需求期间横向扩展，在需求下降时横向缩减。弹性在任何时刻都能提供成本效率和最佳资源使用率。

### AWS 上的容器（Containers）和编排
【容器】提供了一种可靠的方式，将应用程序的代码和依赖项打包成一个单一的、可移植的单元，使其成为需要高安全性、可靠性和可扩展性的工作流的理想选择。其他计算服务
#### 容器和虚拟机
容器将您的应用程序与其运行所需的所有内容打包在一起，因此它在任何计算机上都能保持一致的运行方式。这有助于迁移、更新和管理。容器比虚拟机 (VM) 运行速度更快、更轻便，因为它们共享主机计算机的操作系统。虚拟机通过虚拟机管理程序来运行完整、独立的操作系统，这使得它们在资源使用上效率较低，并且启动时间较长。
![[Pasted image 20251121214053.png]]
【通过容器实现部署一致性】
当开发人员的环境与暂存或生产环境不同时，部署可能会失败，并变得难以调试。容器通过保持应用程序环境的一致性，解决了这一问题，使部署更加顺畅，并有助于进行故障排除。
【通过==编排==扩展容器】
随着容器化应用程序的扩展，管理变得更加复杂。一种从单一主机运行少量容器开始的环境，可能迅速扩展为跨多个主机的数百乃至数千个容器。在这种规模下，手动处理容器生命周期、监控和一般操作变得不可持续。这时，编排工具发挥了作用。它们可以实现部署、扩展和管理的自动化，确保一切顺利运行。

### Amazon Simple Storage Service（Amazon S3）
Amazon S3 是一项对象存储服务，可以在 AWS 云中存储无限量的数据。
Def：Store and retrieve and unlimited amount of data。
- **基本构成和特征**
	- 【S3 对象】：Amazon S3 中的对象是数据存储的基本单元。当您将文件上传到 Amazon S3 时，它会成为一个对象，并在您选择的区域内的多个设施中持久存储。每个 Amazon S3 对象通过其键在存储桶中唯一标识，键本质上是文件名。对象还具有版本 ID、访问控制信息和用户定义的元数据等属性。
	- 【S3 存储桶】S3 存储桶是用于在 Amazon S3 中存储对象的容器。存储桶在整个 AWS 中具有全局唯一的名称，这有助于识别和组织已存储的数据。存储桶作为访问控制的基本单元，可以容纳几乎无限数量的对象。它们在数据管理中发挥着至关重要的作用，能够实现将相关对象分组，并在存储桶级别应用策略。
	- security：
		- 【存储桶策略】：Amazon S3 存储桶策略是_基于资源的策略_，只能附加到 S3 存储桶。S3 存储桶策略指定了对存储桶以及存储桶中的每个对象允许或拒绝的操作。
		- 【基于身份的策略】：控制用户、组或角色可以对 S3 资源执行哪些操作的权限是使用_基于身份的策略_配置的。这些策略直接附加到身份，而不是附加到 S3 资源本身。您可以使用这些策略来指定用户可以访问哪些 S3 存储桶和对象，以及他们可以执行哪些操作。
		- 【加密】：静态加密，传输中加密
- 【AnyCompany Marketing 创建了一个 Amazon S3 存储桶来托管他们新网站的图像。这些图像需要对访问网站的任何人开放，无需身份验证。将图像上传到存储桶后，用户报告说，即使将存储桶策略设置为允许公开访问，他们也无法访问图像。导致此访问问题的最可能原因是什么？】Amazon S3 对象不需要进行标记来使其可公开访问。这是由存储桶策略和访问设置控制的。
### Amazon S3 存储类和 S3 生命周期
- 【S3 Intelligent-Tiering】托管。如果您的数据存在未知的或不断变化的访问模式，这个存储层非常合适。S3 Intelligent-Tiering 将对象存储在三个层中：**频繁访问层、不频繁访问层和归档即时访问层**。
#### 存储分类
- 【S3 ==Standard==】：S3 Standard 被视为云应用程序、动态网站、内容分发、移动和游戏应用程序以及大数据分析的通用存储。当您在未指定存储类的情况下将对象上传到 Amazon S3 时，该对象将默认添加到 S3 Standard 中。
- 【S3 ==Standard== 不频繁访问（Standard-IA）】适用于不频繁访问但在需要时要求快速访问的数据。
- 【S3 ==One Zone== 不频繁访问（One Zone-IA）】与使用三个可用区的 S3 Standard-IA 相比，S3 One Zone-不频繁访问（S3 One Zone-IA）将数据存储在单个可用区中，从而降低了成本。
- 【S3 Express ==One Zone==】S3 Express One Zone 将数据存储在单个可用区中。它的设计专门针对最频繁访问的数据和对延迟敏感的应用程序提供一致的个位数毫秒级数据访问。与 S3 Standard 相比，S3 Express One Zone 提供的数据访问速度最快可提高 10 倍，且请求成本可降低多达 80%。
- 【S3 ==Glacier== Instant Retrieval】S3 Glacier Instant Retrieval 适用于很少访问，但需要在数毫秒内检索的归档数据。此存储类中存储的数据可节省高达 68% 的成本，但延迟和吞吐量性能相同。
- 【S3 ==Glacier== Flexible Retrieval】对于每年访问 1-2 次的归档数据，S3 Glacier Flexible Retrieval 可提供低成本存储。对于备份、灾难恢复、异地数据存储需求，以及偶尔必须在几分钟内检索某些数据的情况，这是一种理想的解决方案。
- 【S3 ==Glacier ==Deep Archive】S3 Glacier Deep Archive 是最低成本的 Amazon S3 存储类。它支持对每年可能访问一次或两次的数据进行长期留存和数字化保存。于存储在 S3 Glacier Deep Archive 存储类中的数据，默认检索时间为 12 小时。
- 【S3 Outposts】Amazon S3 Outposts 通过 Amazon S3 的 API 和功能，向本地 AWS Outposts 环境提供对象存储，支持具有本地数据驻留要求的工作负载。
#### S3生命周期
为避免手动管理对象存储层配置，您可以使用 S3 生命周期配置来自动执行该过程。为一个对象或一组对象定义生命周期配置时，可以选择自动执行以下两类操作：
- _转换操作_：定义对象应在何时转换到其他存储类。
- _过期操作_：定义对象应在何时过期并被永久删除。
例如，您可以在创建对象 30 天后，将对象转换为 S3 Standard-IA 存储类。或者，您可以在创建对象 1 年后，将对象归档到 S3 Glacier Deep Archive 存储类。
## 文件储存 Amazon Elastic File System (Amazon EFS)
### 文件储存 
Amazon EFS 是一项完全托管式可扩展文件存储服务，可用于 AWS 云服务和本地资源。它通过 ==_Linux 网络文件系统 (NFS) 协议_运行==，并在您添加或删除文件时_自动扩展_至 PB 级容量，不会中断应用程序。EFS 的设计支持各种工作负载，可由_多个 EC2 实例同时_访问。
- **Amazon EFS 的优势**
	- 多可用区冗余：Amazon EFS 可在一个区域的多个可用区中自动复制数据，以实现高可用性。这种内置冗余可防止可用区故障，并提供对文件系统的持续访问。
	- 共享访问：Amazon EFS 支持数千个并发 NFS 连接，多个 EC2 实例可以同时访问同一个文件系统。这种共享访问模式使 EFS 成为协作式工作负载和分布式应用程序的理想之选。
	- 弹性存储：当您添加和删除文件时，Amazon EFS 会自动扩缩，无需预置或管理存储容量。而且，您只需为使用的存储付费，这对于不同的工作负载需求都具有成本效益。
- Amazon EFS 存储类：
	- 标准存储类：_EFS Standard_ 和 _EFS Standard-不频繁访问 (Standard-IA)_ 存储类提供多可用区弹性以及最高级别的持久性和可用性。由于可用性和持久性更高，因此成本更高。
	- 单区存储类：_EFS One Zone_ 和 _EFS One Zone 不频繁访问（EFS One Zone-IA）_将数据保存在单个可用区中，可以节省更多成本。与标准 EFS 存储类相比，只需要使用一个可用区，这可以降低存储成本。
	- 归档存储类：_EFS Archive_ 存储类针对每年仅访问几次或更少次数且不需要 EFS Standard 的亚毫秒级延迟的数据进行了成本优化。与 EFS 不频繁访问相比，EFS Archive 的存储价格最多可降低 50%，为很少访问的冷数据提供了更具成本效益的体验。
### Amazon FSx
#### Amazon FSx 的优势
- **文件系统集成**：Amazon FSx 支持行业标准文件系统协议，可方便地与现有应用程序、工作流和开发工具集成。
- **托管基础设施**：Amazon FSx 降低了管理基础设施的复杂性，同时提供了传统文件系统的特性和功能。
- **可扩展存储**：Amazon FSx ==可动态调整资源==，从而避免了复杂的容量规划和手动基础设施管理。
- **经济高效**：Amazon FSx 具有定价模型和自动分层选项，通过仅对已用存储收费，并将不频繁访问的数据转移到成本较低的层来优化成本。

==AWS Storage Gateway ==是一项混合云存储服务，可以将本地环境与 AWS 云存储无缝集成。您可以使用它将本地存储扩展到云，同时保持对常用数据的低延迟访问。
### AWS Storage Gateway
#### 优势
1. 无缝集成
2. 改进的数据管理
3. 本地缓存
4. 成本优化：Storage Gateway==使用云存储进行数据归档、备份和灾难恢复，从而降低本地存储成本==。
#### 网关类型
Storage Gateway 提供三种不同类型的网关，以满足不同的混合存储需求。每种网关类型都设计用于满足特定的使用案例和工作负载要求。
##### Amazon S3 文件网关
当您部署 S3 文件网关时，它会在本地系统中显示为标准文件服务器。写入该服务器的文件会自动上传到 Amazon S3，同时通过智能缓存功能，保持对最近使用的数据的本地访问。
##### 卷网关
使用卷网关，您可以创建虚拟存储卷，同时保持对数据的本地访问权限。它本质上充当本地基础设施与 AWS 云存储之间的桥梁，将云数据以 iSCSI 卷的形式呈现，并通过现有应用程序挂载。
- 缓存卷模式：将主要数据存储在云中，同时将经常访问的数据缓存在本地，以实现低延迟访问。
- 存储卷模式：将在本地保留完整的数据集，同时以异步方式将其备份到云中作为 EBS 快照。
##### 磁带网关
当您部署磁带网关时，它会在备份应用程序中呈现为标准磁带硬件。您的备份软件将数据写入这些虚拟磁带，就像写入物理磁带一样，并将其存储在 Amazon S3 中。

### Amazon Relational Database Service (Amazon RDS)
Amazon RDS 是一项==托管==关系数据库服务，可处理备份、修补和硬件预置等日常数据库任务。Amazon RDS 支持多种类型的数据库实例类，可针对内存、性能或输入/输出 (I/O) 进行优化。

### Amazon Aurora
Aurora 是一种托管式关系数据库，旨在帮助减少不必要的 I/O 操作。它与 MySQL 和 PostgreSQL 兼容，提供高性能和可用性，并可根据工作负载自动扩展。
#### 优势
- 高性能和可用性：Aurora 提供的吞吐量是标准 MySQL 的五倍，是 PostgreSQL 的三倍。它使用跨多个节点的分布式存储系统，提供高性能和高可用性。
- 自动化的存储和备份管理：Aurora 会根据实际数据使用量，自动将存储从 10GB 扩展到 128TB，消除了容量规划中的猜测工作。它还会将数据库持续备份到 Amazon Simple Storage Service (Amazon S3)，支持按时间点恢复。
- 高级复制和容错能力：Aurora 会在三个可用区中复制数据，共保留六份数据副本，==提供 99.99% 的可用性==。它能自动检测数据库故障，并将流量重定向到运行状况良好的副本，确保无数据丢失。
## NoSQL数据库
NoSQL 数据库使用灵活的数据架构，可以存储和检索许多不同类型的信息。在本课中，您将探索 Amazon DynamoDB 这项 NoSQL 数据库服务的优势和使用案例。
- NoSQL 数据库有时被称为==非关系数据库==，因为它们的结构不同于 Amazon RDS 等关系数据库。NoSQL 数据库不使用行和列之间的关系，而是使用_键值对_来构建数据结构。通过==键值对==，数据被组织成由唯一键标识的项。
### Amazon DynamoDB
DynamoDB 是一项完全托管式 NoSQL 数据库服务，可为文档和键值数据结构提供快速且可预测的性能。对于需要灵活架构的使用案例，它是一种功能强大且极其快速的数据库选项，非常适用于对高性能和无缝扩展有需求的应用程序。
